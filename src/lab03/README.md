# Лабораторная работа №3

## 1. Цель работы

Изучение базовых концепций объектно-ориентированного проектирования: наследования и полиморфизма. Разработка иерархии классов на основе абстрактного базового класса (ABC) и реализация уникального поведения для дочерних классов с единым интерфейсом взаимодействия.

---

## 2. Описание реализованной иерархии классов

**Описание:**

*   **Базового класса:**
    Абстрактный класс `Product` описывает общую сущность продаваемого объекта. Он инкапсулирует общие атрибуты (название, базовая цена, скидка, категория, ID, статус активности). В нем реализован расчет базовой математики скидки (`get_final_price`), а также установлены строгие контракты в виде абстрактных методов `process()` и `calculate()`, которые обязывают дочерние классы реализовывать собственную логику обработки и ценообразования.

*   **Дочерних классов:**
    *   `FoodProduct` (Продукты питания) — представляет физический скоропортящийся товар.
    *   `DigitalProduct` (Цифровой товар) — представляет нематериальный продукт в виде электронного файла.
    *   `Service` (Услуга) — представляет работу или мероприятие, привязанное к специалисту.

*   **Различий между ними:**
    *   **Атрибуты:** У еды есть привязка к складу (`stock`) и срокам годности. У цифрового товара склада нет, но есть формат файла и ссылка на скачивание. У услуги нет ни склада, ни файла, но есть привязка к имени исполняющего мастера.
    *   **Ценообразование (Полиморфизм метода `calculate`):** Еда автоматически уценивается на 50%, если срок ее годности истекает через 3 дня (или блокируется для продажи, если просрочена). Цифровой товар рассчитывает наценку с помощью специального словаря коэффициентов на основе формата (например, исходники стоят дороже обычных архивов). Услуга возвращает базовую цену без наценок.
    *   **Обработка заказа (Полиморфизм метода `process`):** Для физического товара (еды) происходит вычитание количества со склада. Для цифрового товара — генерация уникальной ссылки на скачивание файла и имитация отправки на email. Для услуги — простое бронирование времени у специалиста.

## 3. Демонстрация работы

Описание сценариев из `demo.py`:

*   **Сценарий 1: Создание товаров разных типов.** Успешная инициализация физического товара (с истекающим сроком годности), цифрового курса (в формате видео) и услуги. Добавление абсолютно разных объектов в единый абстрактный список — «корзину покупок».
*   **Сценарий 2: Расчет стоимости (Демонстрация полиморфизма).** Программа проходит единым циклом по списку объектов разного типа и вызывает расчет финальной стоимости. Благодаря полиморфизму, для еды автоматически применяется скидка по сроку годности, а для цифрового курса — наценка за формат файла, без использования проверок типа (`if-elif`).
*   **Сценарий 3: Оформление заказа (Обработка товаров).** В едином цикле вызывается абстрактный метод обработки. Хлеб успешно списывается со склада, для цифрового товара генерируется ссылка, а услуга выводит уведомление о бронировании мастера.
*   **Сценарий 4: Состояние товаров после покупки.** Повторный вывод объектов в консоль. Статус цифрового товара изменяется: вместо надписи "Ожидает генерации" отображается готовая сгенерированная ссылка на файл.

## 4. Вывод

Что было изучено:

*   **Наследование:** механизм вынесения общих атрибутов и методов в родительский класс для избежания дублирования кода (DRY). Использование модуля `abc` для создания защищенного абстрактного базового класса, объекты которого невозможно создать напрямую. Применение функции `super()` для безопасного расширения родительского конструктора и методов.
*   **Полиморфизм и утиная типизация:** реализация единого интерфейса (абстрактных методов `process` и `calculate`) для взаимодействия с массивом разнородных объектов. Изучен принцип, при котором вызывающему коду не нужно знать конкретный тип объекта — Питон динамически вызывает правильную реализацию метода на основе фактического класса объекта. Это делает архитектуру программы гибкой и готовой к масштабированию (добавлению новых типов продуктов без изменения логики корзины).
